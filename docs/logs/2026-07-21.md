# 2026-07-21 - 用户与认证（SMTP AUTH）

> 新增 RecipientVerifier。

## 概览

`mail::auth` 三件套（base64 / Password / UserStore / Sasl）、SMTP `AUTH` 动词与 `Session` 集成、`AcceptAllVerifier` 换成用户档驱动的真实校验。

协议细节以 RFC 4954（SMTP AUTH）、RFC 4616（SASL PLAIN）、RFC 4422（SASL 框架）为准；口令哈希用 libsodium Argon2id。

## base64 编解码

`mail::` 顶层新组件（`include/mail/base64.hpp` + `src/base64.cpp`），零 I/O 的纯计算层。

- 放 `include/mail/` 顶层而非 `auth/` 之下：base64/quoted-printable 归属 M7 的 `mime::Encoder`，嵌进 auth 会迫使 M7 反向依赖 auth 模块；
- `base64Decode` 严格：只接受标准字母表（RFC 4648 §4）+ 正确 `=` 填充 + 长度为 4 的倍数，拒绝一切空白（含 CR/LF）、URL 变体 `-`/`_`、多余填充、非零填充位；
- 返回 `bool` 而非 `IoStatus`——结果二元的纯计算，不属"可能部分成功的 I/O"；解码后的 SASL PLAIN 载荷含嵌入 NUL，用 `std::string` 按长度承载。

## `mail::auth::Password` — libsodium Argon2id 封装

`initCrypto` / `hashPassword` / `verifyPassword` / `dummyEncoded` / `scrub`。`<sodium.h>` 只出现在 `src/auth/password.cpp`，绝不进公开头文件，故 mailcommon 的公开接口不泄漏第三方类型。

- 档位固定 **INTERACTIVE**（opslimit=2、memlimit=64 MiB）；MODERATE/SENSITIVE 每次 256 MiB/1 GiB，对每连接认证过重；
- `hashPassword` 产自描述 crypt 串（`$argon2id$` 开头，ASCII，< 128 字节，内含算法/版本/参数/盐/摘要），可原样存进用户档，`verifyPassword` 从串中反解全部参数，无需额外参数列；
- 口令按原始字节处理，不做 SASLprep / Unicode 规范化（RFC 4616 为 RECOMMENDED 非 MUST，Dovecot/Postfix 默认亦不做）；
- `scrub` 内部 `sodium_memzero`（普通 `memset` 可能被优化掉）。

## `mail::auth::UserStore` — 只读用户档

工厂 `open(path)` 一次性载入内存，返回 `Result<std::unique_ptr<UserStore>>`。

#### 用户档格式

```
username:argon2id-crypt-string
```

crypt 串自描述，故无 `{SCHEME}` 前缀；其内含 `$` `,` 但绝不含 `:`，按**首个** `:` 切分安全。用户名 ASCII 大小写不敏感。空行与 `#` 注释行跳过。任一行格式非法（无冒号 / 用户名空 / crypt 空）→ 整个 `open` 失败——用户档是配置，宁可启动即失败也不带残缺账户表上线。

#### 为何返回 `unique_ptr`

有意偏离 `Result<MaildirStore>` 先例：本类持 `std::counting_semaphore`，不可拷贝也不可移动，放不进要求可移动的 `Result<T>`；堆上实例正好可转 `shared_ptr<const UserStore>` 分发给工作线程。

#### 两处安全考量

- **DoS 界**：INTERACTIVE 每次并发哈希占 64 MiB，AUTH 是未认证即可触达的路径，服务端每连接一线程本身不限并发（32 并发即 2 GiB）。`kMaxConcurrentVerifications=4` 的计数信号量把并发 Argon2 数封顶，`GateGuard` RAII 覆盖每条退出路径；
- **计时旁路缓解**：用户不存在时不提前返回，而是拿 `dummyEncoded()` 做一次等价 Argon2 假校验后返回 `Rejected`，使存在（约 100 ms）与不存在两路径耗时一致。`dummyEncoded` 首次调用现算并存函数局部 static，档位常量调整时假串参数自动跟随。

## `mail::auth::Sasl` — 机制对象

`makeSasl(mechanism, store)` 工厂 + `supportedMechanisms()`（=`"PLAIN LOGIN"`）+ `SaslStep{Challenge, Done}` 裁决枚举。

- **职责切分**：Session 负责一切线路格式（base64 编解码、334/235/501 回码、`*` 取消、行长上限），机制对象只见**解码后原始字节**。这让机制可脱离 base64 单测，也让"解码失败 → 501"这条 RFC 规则只存在于 Session 一处。将来加 CRAM-MD5/SCRAM 只改 `sasl.cpp`；
- **PLAIN**（RFC 4616）：`[authzid] NUL authcid NUL passwd`，按**前两个** NUL 切分，密码是第二个 NUL 之后的原始字节、可含嵌入 NUL——绝不当 C 字符串处理；
- **LOGIN**（draft-murchison，IANA 状态 OBSOLETE）：两轮挑战，字符串精确为 `Username:` / `Password:`（每个真实服务器都发冒号形式，非 draft 的 `User Name`）。

## Session AUTH 集成

`SessionConfig` 新增三字段，**全部默认为今日行为**，故既有 17 个 `SessionConfig{}` 调用点逐字节不变：`allowPlaintextAuth`（默认 false）、`requireAuthForMail`（默认 false）、`userStore`（默认 nullptr）。

- **认证状态与事务状态正交**：`authenticated_` / `authenticatedUser_` 是独立成员而非第 5 个 `State` 枚举值。`RSET` 与重复 `EHLO` 只复位事务，不清认证态（RFC 4954 对此沉默，Postfix/Dovecot/Exim 一致做法是认证跨越二者存活）；
- **`runAuth` 子循环**（形如既有的 `collectData`）：驱动 334 挑战 / 响应多轮直到裁决。`AUTH PLAIN`（无初始响应，"给我挑战"）与 `AUTH PLAIN =`（有且为空的初始响应）是不同状态，不可混同；
- **凭据卫生**：`runAuth` 有近十条提前返回路径，`ScrubGuard` RAII 在析构时抹除承载明文口令的 decoded 缓冲，覆盖全部退出路径；
- **EHLO 条件宣告**：仅当启用时追加 `250 AUTH PLAIN LOGIN`，只发 RFC 4954 标准形式；
- **明文 AUTH 默认关闭**：RFC 4954 要求服务器"必须实现一种不允许任何明文口令机制"的配置。本仓 TLS 属 M6，M6 前默认关是这条 MUST 的安全读法。

#### 响应码映射

| 场景 | 回码 |
|---|---|
| 认证成功 | 235 |
| 服务器挑战（空挑战为 `334 ` 带尾随空格） | 334 |
| 未启用 / 机制不识别 | 504（538 已被 RFC 4954 弃用，不发） |
| 已认证再 AUTH / 事务中 AUTH / EHLO 前 AUTH | 503 |
| `*` 取消 / base64 解码失败 | 501（解码失败不谎报成 535） |
| 续行超认证缓冲上限 | 500 5.5.6 |
| 凭据被拒 | 535 |
| 校验暂时不可用（加密库故障） | 454 |
